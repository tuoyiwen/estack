## 1. Loop Engineering：又是啥？企业 Agent 落地的四层工程进化论

- Source: https://www.bestblogs.dev/article/99cc9e2f （腾讯云开发者，李伟山，2026-07-02）
- Source type: 企业 Agent 工程方法论综述（传声筒型，但综合质量高）

### 为什么收

与已有框架互补，构成三视角：Block 六阶段 = 组织成熟度，快手三层重构 = 组织怎么改，Loop Engineering 四层 = 工程栈怎么建（Prompt/Context/Harness/Loop 嵌套关系）。

### 可抽取内容

- **逐层故障诊断框架**（大多数生产级故障在 Harness 层却被误诊为 Prompt/Context 层）→ `$e-innovation-workflow` 诊断问题
- **采纳路径**（夯实 L1/L2 → 重点建设 L3 → 谨慎试点 L4；anti-pattern：跳过 L3 直接 L4）→ workflow-patterns + pilot 检查项
- **L4 三风险**（成本可预测性下降、可靠性新风险面、理解力负债/认知投降）→ governance-controls
- **行业适配差异**（金融 L3 为核心、软工是 L4 原生场景、客服 L1/L2 为核心 L4 有限）→ use-cases
- Harness 五组件（Guides/Sensors/Enforcement/Context Pipeline/Observability）→ technical-enablers

### 注意点

本文是二手综述。四个一手来源值得单独建 source card：Mitchell Hashimoto "My AI Adoption Journey" (2026-02)、OpenAI "Harness Engineering" (2026-02)、Addy Osmani "Loop Engineering" (2026-06)、Anthropic "Effective Context Engineering" (2025-09)。术语时效性强。

## 2. Codex 产品工作重塑访谈（Lenny's Podcast x Andrew Ambrosino）

- Source: https://www.bestblogs.dev/video/6daf60e
- Source type: OpenAI Codex 产品负责人访谈 transcript

### 为什么收

OpenAI Codex 团队本身是 benchmark case：产品工作在 implementation 变廉价后如何重组。同时是博论"末端选择"机制的现成实证素材。

### 可抽取内容

- **Benchmark case**：role collapse（MTS、"角色 = 时间分配的平均"）、zone defense 产品管理、dogfooding loop、10-15 人双位数工程 + 一半设计的团队形态 → benchmark-cases
- **Workflow pattern：产品流程倒置**——旧流程为前置降险（研究/文档/原型都为降低实现成本），新流程为并行探索（90 个原型）+ 策展；"taste/curation 成为贵的部分" → workflow-patterns
- **媒介选择规则**：PRD 没死，实现丰裕时更要选对媒介（求产品清晰用文档，求交互压力测试用原型；警惕过度锚定在看起来 production-ready 的原型上）→ collaboration-patterns（handoff artifact 选择）
- **Research anchor：末端选择 / taste as bottleneck**——implementation 廉价化把人类判断推向策展与选择环节；与工序链框架"研一末端选择"直接对应 → research-anchors
- **"为未来模型先建功能"规划模式**：build ahead、原型入库等模型追上再发布；9 个月计划保持模糊避免虚假精度 → workflow-patterns 或 benchmark case 内

### 注意点

音频转录噪音大，引用用意译。6x 增长、5M WAU 等数字时效性强。

## 4. Token 不经济（腾讯研究院 李刚）

- Source: https://www.bestblogs.dev/article/8f66e3cc
- Source type: token 成本经济学分析（引用多篇 2026 实证研究 + 产业数据）

### 为什么收（P0）

知识库目前没有 token 成本/ROI 治理内容，这是企业客户最先问的问题之一。本文提供了完整的成因分解（定价、技术损耗、场景局限）+ 治理措施 + 现成风险案例（微软收回 Claude Code、Uber 4 个月耗尽全年预算、Meta 撤 Tokenmaxxing 榜）。

### 可抽取内容

- **Token 预算治理与配给制** → governance-controls：配额、审批、成本归因、团队账单、预算池；风险信号 = 消耗与产出不成比例、无意义消耗被激励
- **Token ROI 指标** → metrics-and-rubrics：单位任务 token 成本、无效消耗率（沟通税/熵税占比）、cache 命中率、skill 调用增益、"用最少 token 完成最多工作"作为考核方向
- **Research anchor：编程是通用性的特例** → research-anchors：编程有编译器级确定反馈 + 自动后训练闭环；其他场景验证成本非对称（律师复核 ≈ 从头审阅）。直接呼应快手案例"验证规模化"难题与博论的验证/评价机制。附"token 不经济是技术平权问题"（不懂 Agent 成本动力学的用户处于结构性劣势）
- **技术侧控制** → technical-enablers：语义上下文压缩、模型路由/任务分流、多 Agent 硬预算约束 + 主持人架构、分词器变更风险（闭源分词器黑箱可致 30-50% 成本膨胀）
- **Use-case rubric 增维** → use-cases：候选场景增加"反馈信号可得性 / 验证成本"维度；弱确定性反馈中间地带（影像筛查、需求预测、合同初筛）是桥梁场景
- **对 estack 自身的警示**：SWE-Skills-Bench 数据（79.6% 技能零通过率增益、token 开销最高 +451%；技能效用高度依赖场景匹配）从外部验证了 v0.4 skill 自包含瘦身决定，可在 skills 设计原则里引用

### 不抽取部分

循环融资、私人信贷、电网/民生挤压等宏观金融内容：只随原文归档，不进判断库。

### 注意点

定价数字与厂商事件时效性强，evidence strength 标 partial。三篇被引论文（Salim et al. Tokenomics、Gao et al. SkillReducer、Han et al. SWE-Skills-Bench）值得单独 source card，它们才是 direct 证据。
